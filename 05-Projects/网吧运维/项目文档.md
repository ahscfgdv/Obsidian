### 知识库文档预览

1. 触发阶段 (Frontend UI)
	  在 KnowledgeView.vue 中，当管理员点击“预览”按钮时，会触发 handlePreview(row) 函数：
	-  状态控制：开启 previewVisible 弹窗，并设置 previewLoading = true。
	-  参数传递：将当前行文档的 filename 传给 API 层。
2. API 请求阶段 (Frontend API)
	  调用 api/rag.ts 中的 previewPdf 方法：
	-  核心配置：通过 Axios 发送请求，并显式设置 { responseType: 'blob' }。
	-  作用：告诉浏览器不要将返回的数据尝试解析为 JSON 或字符串，而是保留为原始的二进制大对象 (Blob)。

3. 网络拦截阶段 (Frontend Utils)
	  请求通过 utils/request.ts 的响应拦截器：
	- 类型识别：拦截器检查 config.responseType。
	- 逻辑跳过：识别到是 blob 类型时，跳过常规的业务状态码 (res.code) 校验。因为二进制流中不存在 JSON 结构，直接将原始 Blob 对象返回给业务函数。

4. 后端处理阶段 (Backend API)
	  后端 app/api/v1/rag.py 中的 preview_pdf 接口被调用：
	- 路径定位：根据 filename 在服务器的 knowledge/ 目录下定位物理文件。
	- 流式响应：使用 FastAPI 的 FileResponse 返回。它会自动设置正确的 Content-Type: application/pdf，并将文件内容以数据流的形式发送给前端。

5. 浏览器渲染阶段 (Frontend Browser)  
	  回到 KnowledgeView.vue 的 handlePreview 回调中：
	- URL 生成：使用浏览器原生 API URL.createObjectURL(blob)。这个方法会为内存中的二进制数据生成一个临时的、以 blob: 开头的伪 URL。
	-  组件嵌入：将这个生成的 URL 赋值给 previewUrl。在 HTML 模板中，这个 URL 被绑定到 /<iframe/> 的 src 属性上

 6. 清理阶段 (Memory Management)
	- 资源释放：为了防止内存泄漏，代码在每次生成新预览 URL 之前，以及关闭弹窗时，会调用 URL.revokeObjectURL(previewUrl)。
	-  作用：通知浏览器释放掉之前为那个 Blob 数据分配的内存地址。

# 知识库与 RAG 架构设计说明

  

本文档详细介绍了系统的 RAG (Retrieval-Augmented Generation) 架构设计，包括模块职责、核心逻辑及针对网吧运维场景的优化策略。

  

## 1. 架构设计概览

  

系统采用分层架构设计，确保组件间的低耦合和高可维护性：

  

```text

app/rag/
├── engine/           # 核心引擎层：处理 LCEL 链编排与 AI 逻辑
│   ├── rag_engine.py # RAG 核心链组装，负责检索、格式化与生成
│   └── reranker.py   # 异步并行重排器，利用 LLM 进行二次精排
├── services/         # 业务服务层：封装高层业务逻辑
│   ├── knowledge_base.py # 知识库管理（上传、去重、审核、向量化调度）
│   └── template.py   # 文档模板填充与 PDF 转换
├── loaders/          # 文档解析层：统一的文档加载接口
│   ├── base.py       # 加载器基类
│   ├── pdf.py        # PDF 解析（支持 PyMuPDF + RapidOCR）
│   └── docx.py       # Docx 解析（支持 python-docx + RapidOCR）
├── storage/          # 持久化层：数据存储与检索
│   ├── vector_store.py # ChromaDB 封装，处理向量检索
│   └── history.py    # MySQL 消息持久化，对接 LangChain 历史记录
└── prompts.py        # 提示词中心：统一管理所有场景化 Prompt

```

  

## 2. 核心模块详解

### 2.1 RAG Engine (核心引擎)

- **`rag_engine.py`**:

    - **逻辑设计**: 使用 LangChain Expression Language (LCEL) 构建执行链。
    - **检索策略**: 采用“粗排 + 精排”模式。首先通过向量数据库检索 Top-15 相关片段。
    - **上下文管理**: 自动关联 MySQL 中的历史对话，通过 `HISTORY_CONTEXT_PROMPT` 提供长文本记忆能力。

- **`reranker.py`**:

    - **设计逻辑**: 向量检索仅基于语义距离，可能存在“虚假相关”。我们引入 LLM 对粗排结果进行 0-10 分的打分。
    - **性能优化**: 使用 `asyncio.gather` **并行执行** 15 个文档片段的评分任务，显著降低首字响应时间（TTFT）。

### 2.2 Loaders (解析与 OCR)

- **设计逻辑**: 针对网吧运维文档中常含有截图（如 BIOS 设置、报错弹窗）的特点，所有加载器均继承自 `BaseLoader` 并集成了 **RapidOCR**。
- **功能**: 自动识别文档中的嵌入图像，将图片文字提取并合并至正文，确保图片中的关键报错信息能被索引和检索。


### 2.3 Knowledge Base Service (业务管理)

- **去重机制**: 基于内容计算 MD5。上传时先查 MD5 缓存，避免重复向量化。
- **审核流**:
    1. 用户/管理员提交文档（状态为 `pending`）。
    2. 管理员审核通过后，调用 `audit_approve`。
    3. 系统执行向量化入库，并更新状态为 `published`。
- **检索隔离**: 检索器默认添加 `{"status": "published"}` 过滤器，确保 AI 不会使用未经审核的资料。
  
### 2.4 Storage (存储适配)

- **Vector Store**: 封装 ChromaDB，提供单例模式连接，支持相似度搜索和元数据过滤。
- **History**: 实现 `BaseChatMessageHistory` 接口，将 LangChain 的消息对象透明地持久化到 MySQL `chat_messages` 表中。

## 3. 针对网吧场景的优化逻辑

### 3.1 提示词工程 (Prompt Engineering)

在 `prompts.py` 中，我们为 AI 设定了“资深网吧技术支持专家”的角色，并强制执行以下规则：

- **步骤化回答**: 故障排查必须按 1. 2. 3. 步骤列出。
- **安全第一**: 涉及拆机、放电等操作时，强制提示“断开电源”。
- **真实性保障**: 严禁 AI 幻觉，若知识库无记录，必须明确告知。

### 3.2 混合解析逻辑

网吧运维文档通常由 Word 转 PDF 产生，格式相对统一。我们在解析时保留了页码信息（元数据），并在 AI 回答中关联 `file_url`，方便用户点击预览原始文档进行二次确认。

## 4. 关键技术参数

- **分块策略**: `chunk_size=800`, `chunk_overlap=100`（平衡上下文完整性与检索精度）。
- **粗排阈值**: K=15。
- **精排阈值**: Top-5（最终送入 LLM 上下文）。
- **Embedding**: 使用 Ollama 本地模型（如 `nomic-embed-text`）。

## 5. 维护指南

- **重新索引**: 若向量库损坏，可调用接口删除所有文档，系统会自动从 `Document` 表中的 `source_path` 重新读取并索引。
- **更新 Prompt**: 所有 AI 行为的调整应优先修改 `app/rag/prompts.py`，无需修改引擎代码。
