# Windows Server 远程程序化控制方案指南

本指南旨在解决如何通过代码自动控制远程 Windows 服务器上的软件，涵盖从 GUI 模拟到 API 代理的全套技术栈。

---

## 一、 方案选型矩阵

根据软件特性选择最合适的接入方式：

| **方案名称**     | **核心技术栈**              | **适用场景**          | **稳定性** |
| ------------ | ---------------------- | ----------------- | ------- |
| **API 代理模式** | Python (FastAPI/Flask) | 软件支持 CLI 或有本地 API | 极高      |
| **UI 自动化模式** | Pywinauto / FlaUI      | 必须操作图形界面（GUI）     | 中       |
| **远端执行模式**   | WinRM / SSH            | 执行脚本、管理服务         | 高       |
| **RPA 模式**   | 影刀 / UiBot / Selenium  | 复杂业务流、跨软件操作       | 高       |

---

## 二、 核心方案深度解析

### 1. API 代理模式（推荐方案）

在服务器端运行一个“代理人”程序，将本地操作封装为 Web 接口。

- **实现逻辑**：
    
    1. 在 Server 端部署 Python 环境。
        
    2. 编写 FastAPI 服务监听特定端口。
        
    3. 收到请求后，调用 `subprocess` 执行软件。
        
- **优点**：响应快，支持高并发，易于集成到 AI 机器人系统。
    

### 2. UI 自动化模式（解决“点点点”问题）

针对没有接口、只能靠鼠标点击的软件。

- **库推荐**：
    
    - **Pywinauto**：基于控制台句柄，速度快，不占鼠标。
        
    - **PyAutoGUI**：基于图像识别，万能但易受分辨率影响。
        
- **注意**：此模式需要“活动会话”（详见第三部分）。
    

---

## 三、 关键技术挑战：保持 GUI 会话

**痛点**：当你关闭 RDP（远程桌面）窗口时，Windows 会停止渲染图形界面，导致 UI 自动化脚本报错。

**解决方案**：

1. **禁止断开连接时锁定会话**：
    
    在退出远程桌面时，不要直接点“X”，而是在服务器命令行执行以下命令：
    
    Bash
    
    ```
    # 强制将远程会话转入本地控制台，保持窗口渲染
    tscon.exe %ID% /dest:console
    ```
    
    _(注：需将 %ID% 替换为 `query user` 查到的当前会话 ID)_
    
2. **设置 VNC 替代 RDP**：使用 UltraVNC 或 RustDesk，它们通常能维持虚拟显示器活跃。

## 四、 系统架构建议

1. **指令流**：客户端（如你的微信机器人） -> 发送 JSON 指令。
    
2. **传输层**：通过加密的 HTTPS 或 VPN 隧道传输。
    
3. **执行层**：服务器代理程序解析指令，调用 `pywinauto` 或 `os.system`。
    
4. **状态流**：执行完毕后，截取软件截图或捕获日志，回传给客户端。
